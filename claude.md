## Language Rules
- 설명과 대화 그리고 요청하는 모든 것들 : 한국어, 전문 용어 쓰지 않기(요청자가 이해할 수 있는 말 사용)
- 코드 내부 주석 : 영어
- 설명은 간결, 명확하게
- 플랜부터 먼저 - 바로 만들지 말고 계획서부터
- 수정은 "이거 바꿔줘" 하면 즉시 반영
- API 키는 반드시 .env 파일에 보관 — 코드에 직접 입력 금지, 절대 외부 유출 금지

## 빌드 테스트
- 빌드 : npm run build
- 테스트 (단일) : npm test -- --testPathPattern=<filename>
- 린트 : npm run lint

## 워크플로우
- 코드 수정 후 반드시 타입 체크 실행
- 전체 테스트가 아닌 관련 파일만 테스트 (성능)
- 구현 후 린트 통과 확인

## Core Behavior Rules
1. 사용자의 요청 의도를 먼저 분석
2. 요구사항이 불명확하면 추측하지 말고 질문
3. 기존 코드를 수정할 때는 전체 구조를 먼저 이해
4. 불필요한 변경은 하지 않음
5. 항상 가장 단순하고 유지보수 가능한 방법을 선택

## Code Quality Rules
1. General
- 읽기 쉬운 코드 작성
- 함수는 단일 책임 원칙 준수
- 중복 코드 제거
- 의미 있는 변수명 사용

2. Structure
- 작은 함수로 분리
- 깊은 중첩 피하기
- 에러 처리 명확하게 작성

3. Performance
- 불필요한 연산 제거
- 메모리 낭비 최소화
- 필요할 때만 최적화

## Modification Rules
1. 관련 파일을 먼저 분석
2. 코드 흐름 이해
3. 변경 영향 범위 확인
4. 최소한의 변경만 수행

다음 3가지의 행동은 금지한다.

1. 요청되지 않은 리팩토링
2. 불필요한 라이브러리 추가
3. 프로젝트 구조 변경

## Dependency Rules
1. 반드시 필요한 경우에만 추가
2. 유지보수 상태 확인
3. 프로젝트와 호환성 확인

## Error Handling
1. 에러는 명확하게 처리
2. 사용자에게 의미 있는 메시지 제공
3. silent failure 금지

## Security Rules
1. 민감 정보 하드코딩 금지
2. 입력값 검증 수행
3. 외부 데이터는 항상 신뢰하지 않는다.

## Development Workflow
1. 요구사항 분석
2. 작업 계획 수립
3. 코드 작성 또는 수정
4. 오류 가능성 검토
5. 최종 결과 정리

## Response Format
1. 설명 → 코드 순서로 작성
2. 코드 블록 명확히 표시
3. 필요한 경우 예제 제공
4. 불필요한 장황한 설명 금지

## Special Rules
1. 가짜 API 생성
2. 존재하지 않는 라이브러리 사용
3. 동작 확인되지 않은 코드 제공
4. 모든 코드는 실제로 동작 가능한 형태로 작성해야 한다.
5. 비주얼스튜디오코드가 종료되기 전 항상 모든 대화를 markdown으로 정리해서 claude_session.md 파일로 만들어줘
6. 사용자가 이해하기 쉽도록 코드 설명서를 만들어줘
